## Understanding

T8.8 landed `MixedFastSnapOnlyBuild` as a formal §D.17 BuildFlavor with
compile-time symbol `TDMD_FLAVOR_MIXED_FAST_SNAP_ONLY`. T8.9 wires that symbol
to an actual mixed-precision SNAP kernel — the flavor today compiles cleanly
but still routes SNAP to the FP64 `SnapGpu` class (no behavior change vs
`Fp64ProductionBuild`).

This ships the first FP32-bearing SNAP kernel `SnapGpuMixed` and makes the
`SnapGpuAdapter` compile-time dispatch to it under the SnapOnly flavor.

## Scope decision — Phase A narrow-FP32

Exec pack asks for "bispectrum basis computed in FP32" (aspirational). I am
narrowing to **pair-math FP32 only**, mirroring T6.8a `EamAlloyGpuMixed`
exactly:

FP32 sites:
- `rsq_f = float(rsq)` + cutoff filter on FP32
- `r_f = sqrtf(rsq_f)`, `rinv_f = __frcp_rn` (SFU reciprocal)
- `theta0_f`, `cs_f = __cosf`, `sn_f = __sinf`, `z0_f = r_f · cs_f / sn_f`
- `dz0dr_f` from FP32 composites
- `sfac_f`, `dsfac_f` — FP32 cos/sin on FP32 scaled argument

FP64 sites (unchanged vs snap_gpu.cu):
- Atom positions, forces, per-atom accumulators
- `rootpq`, `cglist`, `beta` tables (kept in device FP64 memory)
- `ulist_r/i`, `ulisttot_r/i`, `dulist_r/i`, `zlist_r/i`, `ylist_r/i`, `blist`
- U-recurrence itself (compute_uarray_device, compute_duarray_device)
- dE/dr dot product (compute_deidrj_device)
- Per-atom PE, per-atom virial
- Host-side Kahan reduction for PE + virial

Rationale for narrowing: bispectrum recurrence (8-level VMK chain at twojmax=8)
compounds FP32 error through dot products with length-O(idxu_max)=165 arrays.
A full-FP32 bispectrum likely exceeds the 1e-5 rel force budget, and the
exec pack's D-M8-14 timeline carve-out ("~4 days") assumes we don't iterate
through bispectrum precision failures. Narrow-FP32 at pair-math is the
risk-minimizing Phase A.

Scientifically: pair-math FP32 → ~1.2e-7 rel on `r`, `rinv`, `sfac`; propagates
through FP64 U-recurrence ~unchanged (FP64 absorbs the FP32 starting value's
ULP); final force error on the order of 1e-6 rel — inside 1e-5 rel budget
with one decade headroom.

Performance: this is modest — SFU sqrt/recip + FP32 trig is ~4-6 ops per
pair in FP32 instead of FP64. Real SNAP bottleneck is the bispectrum
(FP64 in Phase A). Expect 2-8% throughput vs FP64 GPU. If T8.10/T8.11 show
insufficient speedup, a Phase B (FP32 bispectrum) lands as a follow-up
delta with its own SPEC procedure.

## Invariants preserved

1. **Reference sacred**: `Fp64ReferenceBuild` still dispatches to `SnapGpu` (FP64).
   Only `MixedFastSnapOnlyBuild` routes to `SnapGpuMixed`. Preserved by the
   adapter typedef dispatch (compile-time, no runtime branching).
2. **One core, many policies**: `SnapGpuMixed` has byte-identical public
   interface to `SnapGpu`. No architectural fork — just a narrower numeric
   policy on the same kernel structure.
3. **`__restrict__` discipline**: every pointer param in kernels carries
   `__restrict__` (§D.16), matching SnapGpu.
4. **NVTX audit green**: every kernel launch inside `TDMD_NVTX_RANGE`
   scope; `test_nvtx_audit` must still pass.
5. **EAM unchanged under SnapOnly flavor**: `EamAlloyGpuAdapter` must route
   EAM to `EamAlloyGpu` (FP64) under `MixedFastSnapOnlyBuild`, **not** to
   `EamAlloyGpuMixed`. Per D-M8-4 policy: EAM Horner FP32 fails monotonicity,
   keep FP64; the only FP32 intentional contract under SnapOnly is SNAP.
6. **State ownership**: kernels do not mutate positions/types/cells; only
   forces/PE/virial outputs. Matches CPU SnapPotential contract.
7. **No atomicAdd(double\*, double)**: single-block-per-atom layout preserved.

## Risks

1. **Pair-math FP32 exceeds D-M8-8 force budget**: mitigation — if test fails,
   widen scope carefully (e.g., only keep FP32 at `rsq` comparison, FP64 for
   `r`/`rinv`) as a smaller Phase A-lite. NOT blocker, just iteration.

2. **Code duplication**: `snap_gpu_mixed.cu` is 95% identical to `snap_gpu.cu`.
   Accepted — this matches the EAM Mixed precedent (T6.8a did the same).
   Factoring out shared code is a T9/refactor concern, not T8.9.

3. **FMA contraction**: MixedFast flavor builds with `--fmad=true`. This is
   intentional — Philosophy B allows nondeterministic rounding vs Reference.
   Not a risk, just a documented divergence.

4. **EAM adapter ambiguity under SnapOnly flavor**: today `EamAlloyGpuActive`
   dispatches `TDMD_FLAVOR_MIXED_FAST` → `EamAlloyGpuMixed`, else `EamAlloyGpu`.
   `TDMD_FLAVOR_MIXED_FAST_SNAP_ONLY` currently falls into `else` → FP64 EAM,
   which is correct BY ACCIDENT. Explicitly spelling out the check in the
   adapter header removes the accident.
