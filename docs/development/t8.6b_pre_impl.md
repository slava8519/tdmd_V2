**Role:** GPU / Performance Engineer + Physics Engineer (joint).
**Depends on:** T8.6a (scaffolding landed 2026-04-20, commit `5e28c43`), T8.5
(CPU FP64 byte-exact oracle, `test_t6_differential` green, D-M8-7 closed).

## Understanding of the task

Port `SnaEngine` (the CPU FP64 three-pass bispectrum engine landed at T8.4b) to
CUDA. Three GPU kernels matching the three compute passes:

1. **`snap_ui_kernel`** — per-neighbour Wigner-U expansion + per-atom
   `ulisttot` accumulation (CPU analogue: `compute_ui`).
2. **`snap_yi_kernel`** — per-atom Z-list contraction against β coefficients
   → Y-list (CPU analogue: `compute_zi` + `compute_yi`).
3. **`snap_deidrj_kernel`** — per-neighbour dE/dr via Clebsch-Gordan chain
   rule + Newton-3 force scatter (CPU analogue: `compute_deidrj` +
   `compute_duidrj`).

Intermediate helpers (`compute_uarray`, `compute_duarray`, `compute_sfac`,
`compute_dsfac`, `deltacg`) become `__device__` inline functions shared across
the three kernels. Index tables (`idxcg_block`, `idxu_block`, `idxz_block`,
`idxb_block`, Clebsch-Gordan coefficients, `rootpqarray`) are built once on the
host at `SnapGpu::Impl` ctor, flattened to 1D, and uploaded to device once —
then read-only across all compute() calls (same cache pattern as EAM spline
tables at T6.9a).

Bit-exactness vs CPU FP64 path is the T8.7 gate (≤ 1e-12 rel force + PE +
virial). T8.6b ships the kernel body + a functional smoke (not the bit-exact
gate itself — that's T8.7).

## Kernel architecture + thread mapping

### Pass 1 — `snap_ui_kernel`

**CPU analogue.** `compute_ui(int jnum, int ielem)` iterates jj=0..jnum-1,
calls `compute_uarray(x,y,z,z0,r,jj)` then `add_uarraytot(r, jj)` which
accumulates `ulist[jj]` into `ulisttot[atom, u_idx]` scaled by switching
function `sfac(r, rcut)`.

**GPU mapping.** `snap_ui_kernel<<<n_atoms, kThreadsPerBlock>>>`:

- One thread-block = one atom i.
- `kThreadsPerBlock = 128` (warp granularity; matches LAMMPS Kokkos SNAP
  `-pk kokkos gpu/aware on vector 32` default and fits occupancy on sm_80+).
- Each thread-block loops over its atom's neighbours from the CSR cell list
  (`cell_offsets[cell_of_i]..cell_offsets[cell_of_i+1]`, plus 26 stencil
  neighbours — same 27-cell stencil as EAM §7.2).
- For each neighbour jj: compute `x/y/z/r/z0/sfac`; call device `compute_uarray`
  to fill `ulist_r[u_idx]`, `ulist_i[u_idx]` (thread-local stack arrays sized
  `idxu_max` — twojmax=8 → 165 entries → 165×2 doubles = 2.6 KB per neighbour,
  stored in warp-distributed registers/shared).
- Accumulate into shared-memory `ulisttot_r_shared[idxu_max]` /
  `ulisttot_i_shared[idxu_max]` with explicit lane-deterministic ordering
  (§Determinism below).
- After all neighbours processed: flush shared `ulisttot_*_shared[]` to global
  `d_ulisttot_r[atom * idxu_max + u_idx]` / `d_ulisttot_i[...]`.

**Shared memory per block.** `2 × idxu_max × sizeof(double) = 2640 B` (twojmax=8).
Fits comfortably in 48 KB default shared per SM.

**Determinism.** Neighbour accumulation order is atom-local, not global, so
per-atom reduction order is reproducible as long as neighbours are iterated in
canonical order. We use the same sorted-by-id iteration as CPU
`compute_ui(jnum, ielem)` — CPU iterates by `jj = 0..jnum-1` where the
neighbour list is canonicalized by `NeighborListGpu` (T6.4 — bit-exact with
CPU CellGrid already). GPU replicates the exact same `jj` order **per atom**;
intra-thread-block reduction is a **sequential** single-thread accumulation
(no warp-shuffle reduce for Pass 1 accumulation — cheaper to let one lane
drive the loop given shared-memory contention, this matches the CPU ordering
byte-for-byte). This is the "thread 0 of each block does the accumulation"
pattern; other threads help fetch neighbour data but not reduce.

### Pass 2a — `snap_zi_kernel` (internal; not named externally)

**CPU analogue.** `compute_zi()`. Computes per-atom `zlist_r[idxz_max]` /
`zlist_i[idxz_max]` from `ulisttot` via Clebsch-Gordan triple product.

**GPU mapping.** `snap_zi_kernel<<<n_atoms, idxz_max_aligned>>>`:

- One thread = one z-index (one `(j1,j2,j,ma,mb)` tuple).
- `idxz_max` for twojmax=8 ≈ 495; block size = `round_up(idxz_max, 32)` = 512.
- Each thread reads `idxcg_block[j1][j2][j]` (via flat `d_idxcg_block[]`),
  iterates the (ma1,ma2,mb1,mb2) inner loop identical to CPU lines
  sna_engine.cpp:435-503, accumulates `zsum_r` / `zsum_i` in FP64 registers,
  writes to `d_zlist_r[atom * idxz_max + z_idx]` / `d_zlist_i[...]`.
- No reduction across threads — each thread writes its own slot.

**Determinism.** Per-thread FP sequence is the same Clebsch-Gordan inner loop
as CPU; since each thread owns a unique z-slot, no cross-thread reduction
happens. Bit-exact with CPU trivially.

### Pass 2b — `snap_yi_kernel`

**CPU analogue.** `compute_yi(beta)`. Contracts `zlist` against β to produce
`ylist_r` / `ylist_i` of size `idxu_max`.

**GPU mapping.** `snap_yi_kernel<<<n_atoms, idxu_max_aligned>>>`:

- One thread = one u-index (lookup via `d_idxu_block`).
- `idxu_max` = 165 (twojmax=8) → block size 192.
- Each thread reads `beta[k]` for its contracted k and `zlist[atom, z_idx]`
  for the matching z, accumulates in FP64 register, writes
  `d_ylist_r[atom * idxu_max + u_idx]`.
- Inner loop order mirrors CPU `compute_yi` line-for-line (j1/j2/j nested).
- **PE accumulation** — per-atom `pe_per_atom[i] = Σ_k β_k · blist_k(i)` is
  computed here via a parallel reduction that writes `d_pe_per_atom[i]` once
  per atom. Block-level warp-shuffle reduce with canonical lane order
  (lane 0 writes the final sum).

### Pass 3 — `snap_deidrj_kernel`

**CPU analogue.** `compute_deidrj(dedr)` + per-neighbour `compute_duidrj(jj)`.

**GPU mapping.** `snap_deidrj_kernel<<<n_atoms, kThreadsPerBlock>>>`:

- One thread-block = one atom i; threads within the block fan over the atom's
  neighbours.
- Each (thread, jj) pair computes `dedr[3] = Σ over u-indices of
  ylist[u] · dulist[jj, u] · sfac(r_ij) + ylist[u] · ulist[jj, u] · dsfac`.
- Writes force contribution `f[i] += -dedr` and (per Newton-3)
  `f[j] += +dedr` via **deterministic scatter** — NOT `atomicAdd(double*,
  double)` which is forbidden per gpu/SPEC §8.1.
- **Determinism strategy.** Each thread writes **only** to its own atom's
  force slot (thread `i` computes `f[i]` fully from its neighbours'
  contributions — no cross-atom scatter). For the Newton-3 pair `(i,j)`, the
  contribution to `f[j]` is emitted separately by thread `j`'s kernel pass
  against neighbour `i`. This is the same **full-list iteration with
  thread-local writes** pattern as EAM §7.2.3. PE + virial are accumulated
  per-atom and reduced host-side with Kahan (§D-M6-15).

### Host-side reduction

After the three kernels complete, the adapter:

1. D2H's `d_fx` / `d_fy` / `d_fz` (per-atom, 3 × n × 8 B);
2. D2H's `d_pe_per_atom` (n × 8 B);
3. D2H's `d_virial_per_atom` (n × 6 × 8 B);
4. Kahan-reduces host `pe_per_atom[]` → total `potential_energy`;
5. Kahan-reduces host `virial_per_atom[]` → Voigt `virial[6]`.

Per-atom forces are scattered into `AtomSoA::fx/fy/fz` directly (no host
reduction — each atom's force is already complete).

## Memory layout — `Impl` DevicePtr fields

Already pre-declared in T8.6a `snap_gpu.cu`. T8.6b populates these:

**Cached once at ctor (index tables, read-only across all compute() calls):**

```
d_idxcg_block     : int[jdim³]               — flat idxcg_block[j1,j2,j]
d_idxu_block      : int[jdim]                — flat idxu_block[j]
d_idxz_block      : int[jdim³]               — flat idxz_block[j1,j2,j]
d_idxb_block      : int[idxb_max × 3]        — flat (j1,j2,j) per b-index
d_cg_coefficients : double[cg_count]         — Clebsch-Gordan table
d_rootpq          : double[(twojmax+1)²]     — √((p·q)/(p+q)) precomputed
d_radius_elem     : double[n_species]        — Rα
d_weight_elem     : double[n_species]        — wα
d_beta            : double[n_species × (k_max+1)] — β coeffs
d_rcut_sq_ab      : double[n_species²]       — rcut(α,β)² precomputed
```

**Per-atom state (allocated + zeroed per compute() call; sized to current n):**

```
d_ulisttot_r      : double[n × idxu_max]
d_ulisttot_i      : double[n × idxu_max]
d_zlist_r         : double[n × idxz_max]
d_zlist_i         : double[n × idxz_max]
d_ylist_r         : double[n × idxu_max]
d_ylist_i         : double[n × idxu_max]
d_blist           : double[n × idxb_max]
d_pe_per_atom     : double[n]
d_virial_per_atom : double[n × 6]
```

**Per-call inputs (uploaded each compute() — same pattern as EAM at T6.5):**

```
d_types           : int[n]          (or std::byte-typed — matches EAM)
d_x / d_y / d_z   : double[n]
d_cell_offsets    : int[ncells+1]
d_cell_atoms      : int[total_pairs]
```

**Per-call outputs (D2H'd after kernels):**

```
d_fx / d_fy / d_fz : double[n]
```

All allocations via `DevicePool::allocate<T>(stream)`. Pool is cached, so
re-allocations at same size are O(1) after first call (same as EAM).

## Memory budget — W 2000-atom T6 fixture

Per T8.6 pre-impl section on line 40 (twojmax=8):

- Index tables (ctor): ~10 KB total (jdim=9 → jdim³=729 ints each for cg/u/z
  blocks; ~495 doubles cg_coefficients; ~81 doubles rootpq).
- Per-atom state: n × (idxu×2 + idxz×2 + idxu×2 + idxb + 1 + 6) × 8 B
  = 2000 × (165×2 + 495×2 + 165×2 + 56 + 1 + 6) × 8
  = 2000 × 1708 × 8 ≈ **27 MB**.
- Per-call inputs + outputs: ~100 KB (n × few ints/doubles).

Not the ~520 MB figure from the T8.6a pre-impl §"Memory sizing" — that
estimate assumed per-neighbour `ulist_ij` + `dulist_r/i` buffers held globally.
**Thread-block-local stack / shared memory** for those per-neighbour buffers
reduces global footprint by a factor of ~20×. Revised estimate **~27 MB
persistent + ~2 KB shared per block**; comfortably fits on any GPU that runs
EAM at equivalent scale.

Decision: keep per-neighbour `ulist_ij[u_idx]` in thread-block shared memory
(recomputed per neighbour iteration, not retained across Pass 1 / Pass 3).
Pass 3 will re-run `compute_uarray` + `compute_duarray` for each (i, jj) pair
rather than cache the Pass 1 per-neighbour tensors globally. This trades
~2× device FLOPs for 20× device-memory savings — correct tradeoff at T6
scales.

## Explicit assumptions

1. CPU SNAP reference in `src/potentials/snap/sna_engine.cpp` is the locked
   oracle — T8.4b + T8.5 verified byte-exact vs LAMMPS at 1e-12 rel. Any T8.6b
   implementation divergence is a T8.6b bug, not a CPU/oracle issue.
2. `NeighborListGpu` (T6.4) produces the same full-list iteration order for
   SNAP as for EAM — 27-cell stencil, sort-by-id within bucket. T8.6b
   transplants this directly (no custom SNAP neighbour build).
3. Cell-grid cutoff = `SnapData::max_pairwise_cutoff()` already exposed by
   `SnapPotential::cutoff()` (T8.4a landed). `SimulationEngine::init()` sizes
   the cell grid accordingly — no T8.6b work.
4. Index tables (`idxcg_block`, `idxu_block`, `idxz_block`, Clebsch-Gordan
   coefficients, `rootpqarray`) are host-built once by a port of
   `SnaEngine::build_indexlist()` / `init_clebsch_gordan()` /
   `init_rootpqarray()` (no parallel GPU variant needed — sub-millisecond
   host work).
5. `idxu_max` and `idxz_max` at twojmax=8 are small enough (165, 495) to fit
   in static-sized shared-memory stack — no dynamic shared-memory config
   needed. T8.6b hard-codes twojmax=8 support as the M8 target; twojmax=10+
   (MEAM chain) is M9+ work and may require shared-memory bank padding.
6. `--fmad=false` is already the default for `Fp64ReferenceBuild` via
   `cmake/BuildFlavors.cmake` — no T8.6b CMake change needed. The kernel body
   will produce byte-identical FP64 arithmetic to the CPU oracle (modulo
   reduction order absorbed by the 1e-12 rel threshold).
7. `TDMD_NVTX_RANGE` already wraps `snap.compute_stub` scope at T8.6a. T8.6b
   replaces the stub body with three per-kernel ranges:
   `TDMD_NVTX_RANGE("snap.ui_kernel")` / `snap.yi_kernel` / `snap.deidrj_kernel`
   inside a parent `TDMD_NVTX_RANGE("snap.compute")` scope.

## Risks

- **R-T8.6b-1 — reduction-order divergence hits 1e-12 threshold.**
  CPU path accumulates `ulisttot_r/i` sequentially per neighbour; GPU path
  replicates this with single-lane-driven per-block accumulation. If we
  accidentally introduce warp-level parallel reduction for Pass 1
  accumulation, operand ordering differs and 1e-12 rel is at risk. **Mitigation:**
  explicit single-thread accumulation in Pass 1; assert the chosen ordering
  byte-matches a small-atom CPU trace in a debug-print differential (not
  shipped as a test, but used during dev).
- **R-T8.6b-2 — shared-memory size overflows on twojmax > 8.**
  Current static sizing assumes `idxu_max = 165`. If future MEAM chain
  (M9+) pushes twojmax to 10, `idxu_max = 286` → shared footprint doubles.
  **Mitigation:** T8.6b hard-codes a static_assert that `twojmax ≤ 8`; T8.9+
  work revisits if MEAM scale demands. Documented in
  `src/gpu/snap_gpu_kernels.cu` inline comment.
- **R-T8.6b-3 — CUDA compile-time blow-up on nvcc.**
  T8.4b ported 1600 CPU lines in ~30s compile. The same port as nvcc C++-
  device-side may inflate compile to 2-3 min due to `__device__` function
  instantiations + inlining. **Mitigation:** split into 3 TUs (kernels /
  device helpers / tables) not 1 monolithic `.cu`. Separate compilation
  parallelizes on CI compile-only matrix (T6.12).
- **R-T8.6b-4 — LAMMPS GPLv2 block missing in one of the new TUs.**
  The port is verbatim translation — GPLv2 attribution is mandatory on each
  TU. **Mitigation:** pre-commit hook could check for "Copyright (2003)
  Sandia" header in `.cu`/`.cuh` files under `src/gpu/snap_*`; defer the
  hook automation to T8.6b-cleanup if it becomes a review burden, but
  enforce by convention + reviewer check in the T8.6b PR.
- **R-T8.6b-5 — `test_snap_gpu_plumbing` case 3 breaks on post-T8.6b
  compute().** The test expects the sentinel throw. **Mitigation:** rewrite
  case 3 as part of T8.6b (noted in §Tests planned above). Case 1 + 2 + 4
  unchanged.
- **R-T8.6b-6 — NVE drift exceeds 1e-12 rel due to accumulated FP drift.**
  CPU NVE drift on W 64-atom 100 steps is typically ~1e-14 (FP64 round-off
  over ~50-neighbour stencil). If GPU path shows drift > 1e-12, either
  Pass 3 force summation is non-deterministic or the force kernel has a
  real physics bug. **Mitigation:** T8.6b smoke catches this early; if
  >1e-12, halt T8.6b and debug before T8.7 bit-exact gate.
