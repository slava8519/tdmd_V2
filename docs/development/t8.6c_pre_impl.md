# T8.6c pre-implementation report — warp-parallel SNAP GPU kernels

**Role:** GPU / Performance Engineer (primary) + Physics Engineer (byte-exact verification).
**Depends on:** T8.6b (three-kernel functional body, v1.0.18 / commit d1959a7),
T8.7 bit-exact gate (≤ 1e-12 rel force/PE/virial, D-M8-13 closed, commit 19656cf),
T8.10 scout (2026-04-20 — identifies single-lane kernels as the 1.77× gap root cause).
**Unblocks:** M8 ≥20% beat vs LAMMPS (T8.11 cloud-burst gate).

## Understanding of the task

T8.6b shipped a **correctness-prototype** of SNAP GPU: three kernels
(`snap_ui_kernel` / `snap_yi_kernel` / `snap_deidrj_kernel`) each launched as
`<<<n_atoms, 128>>>`, but with the main computation inside `if (tid == 0)`
gates. The 127 non-zero threads per block only perform the parallel
zero-init / shared-memory load prelude, then idle while tid==0 walks the
bispectrum. This is documented at `src/gpu/snap_gpu.cu:645-647`:

> `// Per-atom accumulators — single-lane so we don't need warp reductions.`
> `// (Fine for T8.6b correctness gate; warp-level parallel path is a T8.6c`
> `// perf opt once the byte-exact gate lands at T8.7.)`

T8.7 has landed. **T8.6c distributes the tid==0 inner loops across the
128 threads of each block**, preserving byte-exact output (≤ 1e-12 rel
via deterministic tree reductions — no `atomicAdd(double*)`).

## Invariants to preserve

1. **Byte-exact T8.7 gate (≤ 1e-12 rel).** FP64 reduction order must be
   fully deterministic. Use `__shfl_down_sync` / `__shfl_xor_sync` tree
   reductions within a warp + shared-memory staged reductions across
   warps. Forbid `atomicAdd(double*, double)` (native FP64 atomic on
   sm_120 is supported but reduction order is scheduling-dependent →
   drifts past 1e-12).
2. **Per-atom output array contract.** `d_fx/fy/fz`, `d_pe_per_atom`,
   `d_virial_per_atom` keep identical layout and semantics. T8.6c is a
   kernel-body refactor, not an API change.
3. **Index tables immutable.** `snap_gpu_tables.cu` flatten → upload
   stays unchanged.
4. **`MixedFastBuild` flavor parity.** `src/gpu/snap_gpu_mixed.cu` must
   get the same warp-parallel structure; MixedFast remains FP64 inside
   SNAP per T8.6b (the `--fmad=true` compile-flag delta is the only
   difference vs Reference). Post-T8.6c, both flavors still satisfy T8.7.
5. **NVTX audit green.** Ranges already present around each kernel
   launch; no new H2D/D2H within the kernels.
6. **T6 diff gate (D-M8-8 dense ≤ 9.51e-6 force).** Existing
   DifferentialRunner test for W 2000-atom must stay green per change.
7. **`nelements == 1` M8 scope fence.** SnapGpuAdapter still rejects
   chemflag / quadraticflag / switchinnerflag; warp-parallel refactor
   does not touch the validation path.

## Kernel-by-kernel plan

Block dimension stays 128 threads. Parallelism scope is within the block
across the atom's work, not across atoms.

### `snap_ui_kernel` (5.4 % of runtime — lower priority but consistent)

CPU analogue: `compute_ui(jnum, ielem)` — for each neighbour j, call
`compute_uarray` to expand the Wigner-U recurrence then
`add_uarraytot` to accumulate weighted U into `ulisttot[atom]`.

**Current GPU structure (L283-L340):** tid==0 walks neighbour list, calls
`compute_uarray_device` (single-lane inside helper), then single-lane
adds into shared `ulisttot_r/i`.

**T8.6c structure:**
- Parallelise the **idxu_max sweep** inside `compute_uarray_device`
  (parallel over `jju` index, each thread owns a stride of the U array).
- Neighbour loop stays sequential per block (tid==0 fetches j, then all
  threads cooperate on that neighbour's U-expansion + weighted add).
- Recurrence dependency (`ulist[jju+1]` depends on `ulist[jju]`) forces
  sequential accumulation within a single `j` level, but each `j` level
  has O(j+1)² independent entries parallelisable across threads.
- `add_uarraytot` becomes a strided copy + FMA — naturally parallel.

### `snap_yi_kernel` (33.0 % — high priority)

CPU analogue: `compute_zi` (triple loop j1 ≤ j2 ≤ j over CG coefficients)
→ `compute_yi` (Y_jju = Σ β_k · Z) → B-list inline → PE accumulation.

**Current GPU structure (L449):** tid==0 walks idxz_max entries serially,
each entry computes one Z value via inner CG sum, then scatters β·Z into
ylist. PE accumulated by single-lane.

**T8.6c structure:**
- **Outer parallelism: idxz_max sweep distributed across threads.** For
  twojmax=8, idxz_max ≈ 30-60 entries → parallelisable across warps.
- Inner CG sum (small) stays sequential per thread.
- **Scatter into ylist**: each idxz record writes to a specific ylist_jju
  position. Collisions possible (multiple Z entries contribute to same
  jju). Use warp-level reduction tree + tid-0 consolidation into shared
  ylist_r/i buffer → D2H copy.
- **PE accumulation**: warp-reduce → block-reduce → tid-0 writes to
  `pe_per_atom[i]`.

### `snap_deidrj_kernel` (61.5 % — highest priority)

CPU analogue: `compute_duidrj` + `compute_deidrj` — per atom i, walk
neighbours, compute duarray, contract with ylist[i] for own side and
ylist[j] for peer side, sum into per-atom force + virial.

**Current GPU structure (L686):** tid==0 walks 27-cell neighbour sweep,
for each (i,j) pair runs `compute_uarray_device` + `compute_duarray_device`
plus two `compute_deidrj_device` calls (own + peer), accumulating
fx_acc/fy_acc/fz_acc + virial components by single-lane.

**T8.6c structure:**
- **Parallelise `compute_duarray_device` idxu_max sweep** — same pattern
  as ui kernel (parallel inner recurrence levels).
- **Parallelise `compute_deidrj_device` dot product** over idxu_max —
  each thread owns a stride, warp-reduce partial sums → tid-0 receives
  the three force components per pair.
- **Neighbour walk** stays sequential per block (no cross-neighbour
  races). Per-neighbour work is now ~128-way parallel inside the block.
- **Final per-atom force + virial**: accumulators stay on tid==0 (or
  small warp reduction — they're 9 doubles per atom, cheap either way).

## Risks

1. **Reduction-order drift past 1e-12.** FP64 reductions over varying
   thread counts produce different rounding unless strictly tree-ordered.
   **Mitigation:** use `__shfl_down_sync` with fixed mask and unroll
   power-of-2 reduction; shared-mem stage over 4 warps (128/32) with
   fixed sequence. No data-dependent branches inside the reduction.
2. **Shared memory pressure.** deidrj already allocates
   `10 * idxu_max * 8 B` per block (~13 KB at twojmax=8). Warp-reduce
   workspace adds ≤ 1 KB (4 warps × 3 doubles for force reduction).
   Still well under 48 KB default / 100 KB max on sm_120.
3. **Register pressure.** Adding `tid`/`warp_id` plumbing increases
   live register count marginally. Launch bounds stay open (no
   `__launch_bounds__` currently); if occupancy drops, revisit.
4. **MixedFast divergence.** If T8.6c is landed in `snap_gpu.cu` without
   mirror in `snap_gpu_mixed.cu`, the two flavors drift structurally.
   **Mitigation:** land both in the same commit; tests cover both via
   `SnapGpuActive` typedef.
5. **Debug cost if byte-exact breaks.** Kernel-level reductions are
   harder to debug than single-lane code. **Mitigation:** land one
   kernel at a time (ui → yi → deidrj), verify T8.7 gate after each.
6. **KOKKOS snap/kk comparison.** Real GPU baseline (T8.11 prerequisite,
   LAMMPS KOKKOS build in progress) may reveal T8.6c is insufficient
   — KOKKOS snap/kk is SNL-optimised. In that case T8.6d-style
   block-cooperative-across-atoms work may be needed. **Mitigation:**
   measure after T8.6c; T8.6d scope defined only if gap persists.

## Implementation sequence

Land one kernel at a time, verify byte-exact after each:
1. `snap_ui_kernel` parallelisation (smallest footprint, 5.4 % — dev risk low)
2. `snap_yi_kernel` parallelisation (33 % — medium complexity, tree reductions)
3. `snap_deidrj_kernel` parallelisation (61.5 % — largest, most complex)
4. Mirror all three to `snap_gpu_mixed.cu`.
5. SPEC §7.5 update + RESULTS.md post-T8.6c measurement.

Each commit: byte-exact test + functional smoke + T6 diff. No commit
land if any of these fail.
