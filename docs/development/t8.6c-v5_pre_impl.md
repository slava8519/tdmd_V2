# T8.6c-v5 pre-implementation report — per-bond dispatch SNAP GPU kernels

**Role:** GPU / Performance Engineer (primary) + Architect (D-M8-13 invariance review) + Physics Engineer (byte-exact re-validation at each stage).
**Depends on:** T8.6c-v4 (commit `223a35a`, intra-layer (mb,k) parallel recurrence — last single-GPU lever before architectural rewrite); T8.7 bit-exact gate (D-M8-13 ≤ 1e-12 rel force/PE/virial, commit `19656cf`); T8.9 MixedFast narrow-FP32 (D-M8-8 ≤ 1e-5 rel force).
**Unblocks:** M8 ≥20% gate pursuit OR honest "why not" documentation (master spec §14). Post-v5 the single-GPU lever is exhausted — remaining path is multi-rank TD (T8.11).

## Understanding of the task

Post-T8.6c-v4 (2026-04-20) parity gap vs LAMMPS KOKKOS `snap/kk` on 2000-atom BCC W / RTX 5080:

| Kernel | TDMD ms/step | LAMMPS equiv ms/step | Gap |
|---|---:|---:|---:|
| `snap_ui_kernel` | 7.54 | Neigh+CK+PreUi+Ui+TransformUi = 0.47 | **16.0×** |
| `snap_yi_kernel` | 11.52 | ComputeYi + BetaLinear = 1.36 | **8.5×** |
| `snap_deidrj_kernel` | 35.57 | ComputeFusedDeidrj + ComputeForce = 2.29 | **15.5×** |
| **Σ GPU kernels** | 54.63 | 4.12 | **13.3×** |

T8.6c-v4 closed the intra-block recurrence parallelism. The remaining gap is **outer dispatch shape**:

- Today TDMD launches `<<<n_atoms, 128>>>` = 2000 blocks. Each block walks its atom's 3×3×3 cell stencil and processes its ~50 neighbours **sequentially** (bond-by-bond inside the inner loop). T8.6c-v4 parallelised the work inside one bond's recurrence across 128 threads.
- LAMMPS KOKKOS launches a TeamPolicy with **per-bond teams** (~100K teams for 2000×50-neighbour system), each of 4–8 threads. All bonds of the whole system are in flight simultaneously across 84 SMs.
- RTX 5080 has 84 SMs × 1536 max-resident-threads = 129 K max concurrent threads. TDMD's 2000-block × 128 = 256 K threads spec already saturates but with very coarse-grained work per block. LAMMPS's 100K-block × 4-thread spec saturates with fine-grained bond-level work, giving better latency hiding and SIMT utilisation.

**T8.6c-v5 distributes the bond loop across the grid**, turning per-atom-with-inner-loop into per-bond-with-per-atom-gather. Per-atom ownership of output arrays (`ulisttot`, `dedr`, forces) is preserved through staging — no `atomicAdd(double*, double)`, as required by gpu/SPEC §6.1.

### Why "reduce-then-scatter" and not atomics

gpu/SPEC.md §6.1 explicitly forbids `atomicAdd(double*, double)` without sort (race-order dependency breaks FP64 bit-exactness past 1e-12). LAMMPS KOKKOS accepts run-to-run FP64 drift; TDMD's T8.7/D-M8-13 ≤ 1e-12 rel gate does not.

Permitted alternative (SPEC §6.1 explicit): **reduce-then-scatter** — per-bond kernels write to per-bond staging arrays (exclusive, no conflicts), followed by per-atom gather kernels that read bond ranges in deterministic order (matching CPU's sequential loop order). The gather kernel is exactly the T8.6c-v3-style warp-shuffle tree reduction extended from per-atom's neighbour stencil to per-atom's bond range.

## Relevant spec sections

- **master spec §14 M8** — SNAP proof-of-value gate. Post-v5 either ≥20% faster on ≥8 ranks, or honest documentation (master spec §14 M8 also mandates honest failure path).
- **gpu/SPEC.md §6.1** — forbidden: `atomicAdd(double*, double)` without sort. Permitted: reduce-then-scatter via sorted bond list. This is the load-bearing constraint.
- **gpu/SPEC.md §6.2** — Kahan compensation intra-kernel. Per-atom gather must Kahan-accumulate over bond range. Already used elsewhere (neighbor_list_gpu, EAM density accumulator).
- **gpu/SPEC.md §7.5 + §8.1** — FP64-everywhere Reference path, `--fmad=false` on Fp64ReferenceBuild. Per-bond Ui / Deidrj kernels must use FP64 accumulators in Reference; MixedFast inherits T8.9 narrow-FP32 only on `sqrtf` (all bispectrum math stays FP64).
- **gpu/SPEC.md T8.7 acceptance (§7.5 "T8.7 acceptance")** — ≤ 1e-12 rel force/PE/virial CPU FP64 ↔ GPU FP64 on W 2000-atom. Must be regression-green at every stage gate. Non-negotiable.
- **gpu/SPEC.md T8.9 acceptance (D-M8-8)** — ≤ 1e-5 rel force, ≤ 1e-7 rel PE on MixedFast dense cutoff. Preserved.
- **master spec §D.16** — `__restrict__` on all hot-kernel pointer params. New kernels follow convention.
- **master spec §D.17** — no new BuildFlavor created. T8.6c-v5 is a kernel-body refactor per existing flavors (Fp64Ref, MixedFast, MixedSnapOnly).
- **gpu/SPEC.md §9 + D-M6-14** — `TDMD_NVTX_RANGE` per kernel launch. All new kernels get their own ranges.

## Invariants to preserve

1. **T8.7 byte-exact gate ≤ 1e-12 rel** — GPU FP64 ≡ CPU FP64 on W 2000-atom. The bond-list pre-pass emits bonds in CPU loop order (3×3×3 cell stencil, same iteration order) — gather kernels sum in that order via Kahan. Preserves FP64 reduction order.
2. **D-M8-8 MixedFast ≤ 1e-5 rel force, ≤ 1e-7 rel PE** — FP32 narrow still scoped to `sqrtf` only.
3. **No `atomicAdd(double*, double)`** anywhere. Per-bond writes are exclusive (bond_idx is unique per team). Per-atom writes from gather are exclusive per block. Forbidden-pattern audit stays green.
4. **`SnapGpu::compute()` public API unchanged.** All changes are internal to `Impl`. Adapter (`snap_gpu_adapter.cpp`) untouched.
5. **Three-flavor parity.** `snap_gpu.cu` (Reference + Production), `snap_gpu_mixed.cu` (MixedFast, MixedSnapOnly) get parallel rewrites. Shared device helpers in `snap_gpu_device.cuh`.
6. **NVTX audit green.** New ranges for bond-list-build, per-bond-Ui, Ui-gather, per-bond-Deidrj, force-gather. Existing ranges renamed for clarity.
7. **`nelements == 1` M8 scope fence.** SnapGpuAdapter's chemflag/quadraticflag/switchinnerflag rejection path unchanged.
8. **Newton-3 convention preserved.** Forces computed own+peer via Newton-3 double-eval, same as today. Per-bond storage stores dedr_own and dedr_peer separately per (i, j) bond so gather can apply sign convention cleanly.
9. **Thermostat/integrator coupling intact.** `d_fx/fy/fz` additive-write semantics preserved at gather stage (caller pre-seeds `+=` as today).

## Architectural design

### Pipeline shape (T8.6c-v5)

```
  [pre-pass]                             [per-bond]                  [per-atom]
  ─────────────────────────────────────────────────────────────────────────────
  build_bond_list_kernel        →   (per-bond Ui contributions)  →   ui_gather_kernel
   <<<n_atoms_padded, 128>>>           snap_ui_bond_kernel              <<<n_atoms, 128>>>
                                       <<<n_bonds, 128>>>

                                   snap_yi_kernel (unchanged)
                                   <<<n_atoms, 128>>>

                                   snap_deidrj_bond_kernel      →    force_gather_kernel
                                   <<<n_bonds, 128>>>                 <<<n_atoms, 128>>>
```

### Storage additions

New per-bond arrays (allocated in `SnapGpu::Impl::prepare()` once per neighbour-list rebuild):

| Array | Size | Layout | Writer | Reader | Purpose |
|---|---|---|---|---|---|
| `d_bond_i[]`, `d_bond_j[]` | `n_bonds` | SoA uint32 | build_bond_list | all bond kernels | atom indices |
| `d_bond_dx/dy/dz[]`, `d_bond_rsq[]` | `n_bonds` | SoA double | build_bond_list | Ui_bond, deidrj_bond | pair geometry |
| `d_bond_type_i[]`, `d_bond_type_j[]` | `n_bonds` | SoA uint8 | build_bond_list | Ui_bond, deidrj_bond | per-bond types |
| `d_atom_bond_start[]` | `n_atoms+1` | uint32 offsets | build_bond_list | both gather kernels | CSR offsets (i → bond range) |
| `d_ulist_bond_r/i[]` | `n_bonds × idxu_max` | row-major double | Ui_bond | Ui_gather | per-bond Ui contribution |
| `d_dedr_bond_own[][3]`, `d_dedr_bond_peer[][3]` | `n_bonds × 3` double | SoA | deidrj_bond | force_gather | per-bond dedr Newton-3 own/peer |
| `d_virial_bond[][6]` | `n_bonds × 6` double | SoA | deidrj_bond | force_gather | per-bond virial contribution |

**Memory budget for 2000-atom W BCC (twojmax=8, ~50 neighbours/atom → n_bonds ≈ 100 K):**
- ulist_bond: 100 K × 55 positions × 2 × 8 B = **88 MB** (dominant)
- dedr_bond: 100 K × 6 × 8 B = 4.8 MB
- bond_geom: 100 K × 8 × 8 B = 6.4 MB
- **Total ~100 MB** on 16 GB GDDR7 — fits with room for T4 Ni-Al scale-up.

For larger systems (1 M atoms × 50 neighbours = 50 M bonds × 88 B ulist contribution = **44 GB ulist_bond alone**) the per-bond ulist staging is memory-prohibitive. This motivates **stage-2a fallback** (Option E from scout) for very large systems, deferred to post-v5 if needed.

### Determinism strategy — CPU-order bond emission

CPU SNAP (`SnaEngine::compute_ui` in `src/potentials/snap/sna_engine.cpp`) iterates neighbours in the cell-stencil walk order defined by `NeighborList::neighbors_of(i)`. GPU today walks the same 3×3×3 stencil with `for (dz∈-1..1) for (dy∈-1..1) for (dx∈-1..1) for (k∈cell_atoms)`.

**Bond emission order (build_bond_list kernel):** each atom i emits bonds in the identical 3×3×3 × cell_atom_k order that today's snap_ui_kernel uses. Per-atom bond ranges `[atom_bond_start[i], atom_bond_start[i+1])` preserve this order. Gather kernels sum in this order via Kahan accumulator.

**Correctness claim:** per-atom gather sum for atom i sums `ulist_bond[bond_idx]` over bond_idx ∈ [start[i], end[i]) in the same sequence CPU's `compute_ui` accumulates `ulist` per-neighbour. FP64 add is not associative but IS deterministic for a fixed order → bit-exact with CPU.

### Per-bond kernel shape

`snap_ui_bond_kernel<<<n_bonds, 128>>>`:
- Grid: `bondIdx = blockIdx.x`
- Block: 128 threads, reuse T8.6c-v4 `compute_uarray_parallel_device` verbatim
- Work: read `bond_i[bondIdx], bond_j[bondIdx], bond_dx/dy/dz, bond_rsq, types`; compute Cayley-Klein parameters (tid==0); execute `compute_uarray_parallel_device` cooperatively; write weighted `sfacwj · ulist[k]` → `d_ulist_bond_r/i[bondIdx*idxu_max + k]`
- No global Ui accumulation — that happens in gather.

`snap_deidrj_bond_kernel<<<n_bonds, 128>>>`:
- Same shape. Reads `d_ylist[bond_i]` and `d_ylist[bond_j]` (via global load — ylist is per-atom, read-only in this kernel).
- Executes `compute_duarray_parallel_device` cooperatively, reduces dedr_own and dedr_peer via warp-shuffle + block-reduce (T8.6c-v3 technique, unchanged).
- Writes `d_dedr_bond_own[bondIdx]`, `d_dedr_bond_peer[bondIdx]`, `d_virial_bond[bondIdx]`.

### Per-atom gather kernels

`snap_ui_gather_kernel<<<n_atoms, 128>>>`:
- Grid: atomIdx = blockIdx.x
- Block: reads `atom_bond_start[atomIdx..atomIdx+1]`; loops bonds in-order; accumulates `ulist_bond[bondIdx, k]` → shared-memory Kahan accumulator per k ∈ [0, idxu_max)
- Seeds diagonal (CPU `zero_uarraytot` + wself identity) before bond loop
- Writes `d_ulisttot_r/i[atomIdx, k]`
- This stage is per-atom-sequential over bond range (matches CPU order), parallel over k (128 threads / 55 positions = 2.3 positions per thread batch).

`snap_force_gather_kernel<<<n_atoms, 128>>>`:
- Reads `atom_bond_start[atomIdx..atomIdx+1]`; for each bond computes `F_i += dedr_own - dedr_peer` (Newton-3), virial += dedr_own · Δr, PE contributions already in yi kernel
- Writes `d_fx/fy/fz[atomIdx] +=`, `d_virial_per_atom[atomIdx*6..]`
- tid==0 write (single-lane final sum — small, 3 components × atoms), 128 threads parallel over bonds within atom

## Staging plan

Each stage is an independent commit with its own byte-exact green T8.7 gate before moving to the next. Rollback per stage is trivial (single-commit revert).

### Stage 1 — bond list infrastructure + pre-pass kernel (no kernel migration yet)

- New file: `src/gpu/snap_bond_list_gpu.cu`
- Kernel: `build_bond_list_kernel` (per-atom pass, emits bonds to SoA in CPU order, with exclusive scan for CSR offsets)
- Host side: `BondListHandles` struct owning device pointers, allocated once per neighbor rebuild, reused across steps
- Integration: called from `SnapGpu::Impl::prepare()` or at start of `compute()`
- **Validation:** unit test `tests/gpu/test_bond_list_matches_cpu_stencil_order.cpp` — for each atom, assert bond_list iteration order matches CPU `SnaEngine`'s neighbour enumeration
- **Expected wall-clock impact:** +0.2–0.5 ms/step (one-shot per neighbor rebuild, not per-step)

### Stage 2 — Ui per-bond + gather (replaces snap_ui_kernel)

- Add `snap_ui_bond_kernel` and `snap_ui_gather_kernel` in `snap_gpu.cu`
- Keep old `snap_ui_kernel` under `#if TDMD_SNAP_LEGACY_PERATOM` define for A/B rollback
- Update `SnapGpu::Impl::compute()` to dispatch new pair
- MixedFast mirror in `snap_gpu_mixed.cu`
- **Validation gate:** T8.7 byte-exact ≤ 1e-12 rel on W 2000-atom; D-M8-8 ≤ 1e-5 rel on MixedFast. `tests/gpu/test_snap_gpu_bit_exact.cpp` and `test_snap_mixed_fast_within_threshold.cpp` must be green.
- **Expected per-kernel speedup:** Ui 7.54 → ~0.5–1.0 ms/step (kernel time; gather adds ~0.3 ms; net 2–4× Ui alone, 1.3–1.5× total step)

### Stage 3 — Deidrj per-bond + force gather (replaces snap_deidrj_kernel)

- Add `snap_deidrj_bond_kernel` and `snap_force_gather_kernel`
- Old `snap_deidrj_kernel` under `#if TDMD_SNAP_LEGACY_PERATOM`
- Dispatch updated
- MixedFast mirror
- **Validation gate:** T8.7, D-M8-8 green + EAM/SNAP M5 anchor test (if reachable in single-GPU) + 100-step NVE stable integration ≤ 1e-5 drift
- **Expected speedup:** Deidrj 35.57 → ~2–4 ms/step (kernel) + ~0.5 ms gather; 3–5× total step

### Stage 4 — consolidation + legacy removal

- Remove `#if TDMD_SNAP_LEGACY_PERATOM` guards and old kernels (keep git history as reference)
- Update `gpu/SPEC.md` §7.5 change-log with v5 final numbers
- Update `potentials/SPEC.md` SnapPotential section with per-bond architecture note
- Update `RESULTS.md` in `verify/benchmarks/t6_snap_tungsten/scout_rtx5080/`
- **Expected final numbers (pessimistic):** MixedFast 92 → 8–12 ms/step = ~2× from LAMMPS KOKKOS (parity-ish)
- **Expected final numbers (optimistic):** MixedFast 92 → 5–7 ms/step = ~1.5× from LAMMPS KOKKOS (close to parity)
- **If M8 ≥20% not met:** honest doc per master spec §14 — go to multi-rank TD scout (T8.11)

### Stage 5 (optional, if v5 has headroom) — snap_yi warp-tuning

- LaunchBounds<256, 2> to match LAMMPS occupancy target
- Better ulisttot coalescing (contiguous reads per (atom, z-idx))
- **Expected speedup:** yi 11.52 → ~3–4 ms/step (already good after T8.6c-v2; 8.5× gap hard to close without rethinking Z-contraction)

## Risks

1. **Memory blow-up on larger systems.** ulist_bond is ~44 GB for 1 M atoms. **Mitigation:** document as v5 known limit; add pre-check in `Impl::prepare()` that rejects allocations > 80% of device memory with a scientist-actionable error pointing to "use smaller zone or wait for v6 streaming variant". Alternative (deferred): chunked processing — process K bonds at a time with rolling buffers.
2. **Bond list rebuild frequency.** Currently neighbour list is rebuilt when skin is exceeded. Bond list must rebuild on same trigger. **Mitigation:** piggyback on existing `NeighborList::rebuild()` signal; add bond_list_rebuild as a subclass of the same telemetry NVTX range.
3. **Byte-exact failure at Stage 2 or 3.** If reduction order in gather kernel doesn't match CPU, force rel can be 1e-11 or worse. **Mitigation:** unit test at Stage 1 compares bond emission to CPU neighbor iteration. Stages 2 and 3 keep legacy kernels behind ifdef so green baseline is always accessible during debugging.
4. **Per-bond kernel occupancy shortfall.** 100K blocks might still be kernel-register-limited. **Mitigation:** NCU/nsys profile at Stage 2 to confirm active blocks per SM; if low, reduce threads-per-block from 128 to 64 or restructure recurrence to reduce register pressure.
5. **Gather kernel is serial over bonds-per-atom (~50 bonds).** This is the new bottleneck after per-bond work is parallelised. For 2000 atoms × 50 bonds = 100K sequential adds total = 1.2K per SM sequential at 84 SMs. Kahan adds ~4 flops per add = 5K cycles per block = 1.5 μs per atom × 2000 atoms = 3 ms worst-case per gather. **Mitigation:** acceptable overhead (7% of step). If problematic, extend to tree reduction over bond range with log(n_bonds) levels.
6. **MixedSnapOnly regression.** T8.6c-v4 revealed T8.9 narrow-FP32 `sqrtf` setup path dominance. Per-bond-dispatch kernel runs this setup once per bond in parallel rather than sequentially — will show further MixedSnapOnly improvements, not regressions.
7. **2–3 day focused work estimate might slip.** Each stage has its own byte-exact gate; hitting one gate could take a full day of debugging FP64 ordering issues. **Mitigation:** ifdef legacy fallback at each stage; if Stage 2 gate blocks progress >1 day, post partial results and assess whether Stage 3 is worth continuing or whether honest "why not" doc wins.

## Rollback strategy

At any stage, `git revert HEAD` restores the previous T8.6c-v4 baseline. The `#if TDMD_SNAP_LEGACY_PERATOM` define in Stages 2 and 3 allows runtime A/B testing before Stage 4 consolidation; a failure discovered post-Stage 2 doesn't block Stage 3 proceeding from the legacy baseline.

2. D-M8-8 ≤ 1e-5 rel force, ≤ 1e-7 rel PE green (MixedFast W).

4. No new NVTX gaps; forbidden-pattern audit green (no `atomicAdd(double*, double)`).
5. SPEC.md v1.0.22 → v1.0.23 with v5 final numbers; RESULTS.md updated.
6. Memory snapshots updated with post-v5 headline.

## Post-v5 path

- If ≥20% M8 gate hit: document and celebrate, proceed to T8.11 cloud-burst multi-rank scout.
- If parity (~1×) but not ≥20%: documented honestly per master spec §14, pivot to T8.11 multi-rank TD as primary gate-closer (TD halo-vs-all-to-all is TDMD's native advantage).
- If >2× from LAMMPS KOKKOS (partial failure): retrospective on what blocked per-bond dispatch (register pressure? memory BW? kernel launch overhead?); evaluate whether post-v5 Stage 5 (yi tuning) or deferred v6 (streaming bond chunks for larger systems) is worth pursuing before multi-rank.
