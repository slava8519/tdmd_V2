# T8.10 — GPU SNAP perf scout TDMD vs LAMMPS on RTX 5080 — pre-impl report

**Scope (user explicit):** "замер GPU SNAP perf vs LAMMPS на RTX 5080, dev-only baseline".
Dev-only — NOT the formal cloud-burst T8.11 gate. This is the **pre-flight
measurement** the user needs before approving cloud budget for T8.11.
Project-level decision input: is TDMD GPU SNAP anywhere near LAMMPS on a
single consumer GPU?

**Role:** GPU / Performance Engineer
**Depends:** T8.9 (DONE — MixedFastSnapOnlyBuild narrow-FP32 SNAP path
landed; all 3 flavors green)
**Blocks:** T8.11 cloud-burst decision (go/no-go based on scout numbers)

## Why this, why now

Memory `project_m8_perf_smoke` already records CPU-side comparison: TDMD 30%
slower than LAMMPS on single-rank 2000-atom SNAP. Multi-rank SNAP does not
scale yet. The **only credible M8 ≥ 20% speedup path is GPU** (T8.6 → T8.7
→ T8.9). This scout answers: does that claim survive empirical test on
local hardware (RTX 5080), before spending cloud budget on T8.11?

**Three plausible outcomes:**
- **(a) TDMD GPU SNAP competitive** (≤ 30% gap or better) → T8.11 cloud burst
  is justified; proceed with formal fixture + scaling matrix.
- **(b) TDMD GPU SNAP significantly slower** (>> 2×) → investigate before
  cloud burst; T7.8b-style algorithm / kernel tuning first.
- **(c) TDMD GPU SNAP faster** → unlikely but would reshape M8 artifact gate
  write-up materially.

All three outcomes warrant recording as a documented dev-only baseline.

## Invariants to preserve

1. **Same fixture, same potential, same thermostat.** Compare apples-to-apples:
   2000-atom BCC W, W_2940_2017_2 SNAP coefficients (the canonical T6
   fixture, Wood & Thompson 2017), metal units, dt=0.5 fs, NVE, 1000 steps.
2. **Same hardware, back-to-back runs.** Local RTX 5080, cold-start cache-
   cleared; no thermal throttling skew (cool GPU between runs if needed).
3. **Warmup amortisation.** First ~50–100 steps drop (JIT / kernel compile /
   CUDA context); report steady-state wall-clock per step.
4. **Honest methodology documentation.** This scout is informational — not
   a formal artifact gate. Any asymmetry (e.g. LAMMPS neighbor rebuild
   frequency vs TDMD skin policy) must be documented, not hidden.
5. **Existing regression tests remain green.** The scout adds scripts +
   results — does not modify any tested code path.

## Risks

- **Warmup / kernel-JIT noise.** RTX 5080 is sm_120, PTX JIT cost non-
  trivial on first CUDA context. Mitigation: report median of 3 runs
  after first-run discard; document this in RESULTS.md.
- **LAMMPS GPU package configuration.** Default `-pk gpu 1` uses single
  GPU; LAMMPS SNAP GPU is known to be slower than CPU-parallel SNAP at
  modest atom counts (< 10k). Document this — it's a fair comparison
  axis, not a bug.
- **Neighbor-list rebuild cadence.** LAMMPS and TDMD use different skin
  policies; identical timestep does not guarantee identical rebuild
  frequency. Mitigation: extract rebuild count from LAMMPS log.lammps +
  TDMD telemetry and report both; note any divergence.
- **Thermal throttling on consumer card.** RTX 5080 may throttle under
  sustained load. Mitigation: 1000-step runs are short enough that
  throttle is unlikely; still, monitor GPU temp between runs.
- **Consumer-GPU ≠ data-center-GPU.** RTX 5080 ≠ A100/H100 (cloud
  target). Scout tells us the *gap ratio* on this hardware; T8.11 will
  re-measure on cloud hardware where ratio may differ (A100 is 3 TB/s
  HBM vs RTX 5080 ~960 GB/s GDDR7; SNAP is compute-bound so ratio
  should be similar — not perfectly transferable but directionally
  valid).

## Methodology — wall-clock capture

For each config × each flavor:
1. Pre-run: sleep 30 s between runs (thermal settle); drop pagecache if needed.
2. Run: `time tdmd run --timing config.yaml` (3 repeats); similarly
   `time lmp -in lammps_script_gpu.in` (3 repeats; `-sf gpu -pk gpu 1`).
3. Extract wall-clock, steps/sec, pair time breakdown.
4. Record median.

Metrics:
- **Total wall-clock (s)** — apples-to-apples primary number.
- **Steps/sec** — throughput.
- **Atom-steps/sec** — 2000 × steps/sec.
- **Pair time fraction** — how much runtime was SNAP force calc.
- **Neighbor rebuild count** — from telemetry / log.lammps.
