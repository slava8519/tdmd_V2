# T8.6 pre-implementation report — SnapPotential GPU FP64

**Date:** 2026-04-20 · **Author session:** post-T8.5 closure · **Role:** GPU /
Performance Engineer + Physics Engineer (joint, per m8 exec pack T8.6).

## Understanding of the task

Port `SnapPotential` CPU FP64 force body (T8.4b) to CUDA so that
`runtime.backend: gpu + potential.style: snap` produces forces on device with
D-M8-8 bit-exact gate vs CPU FP64 (≤ 1e-12 rel per-atom). Depends: T8.5 CPU
oracle locked.

## Source sizing — why this is a major port

| CPU asset | Lines | Role |
|---|---|---|
| `src/potentials/snap/sna_engine.cpp` | 1259 | Verbatim LAMMPS USER-SNAP `sna.cpp` port |
| `src/potentials/snap.cpp` | 279 | Outer-loop + half-list→full-list bridge |
| `src/potentials/include/tdmd/potentials/snap/sna_engine.hpp` | 201 | Engine class decl |
| `src/potentials/include/tdmd/potentials/snap.hpp` | 93 | Potential adapter |

`SnaEngine` holds **~30 member fields** (pointer-arrays `double**`,
pointer-pointer-arrays `double***` for `idxcg_block`, Clebsch-Gordan tables,
Wigner U-function scratch `ulist_r_ij[nmax][idxu_max]`, etc.). The LAMMPS
`memory->create` pointer-array idiom does not translate directly to `__device__`
code — each `T**`/`T***` needs flattening to contiguous `T*` with manual
stride arithmetic on device.

For twojmax=8 (T6 fixture): `idxu_max ≈ 165`, `idxcg_max ≈ 495`, `idxz_max ≈
495`, `idxb_max = 56`. Per-atom state ≈ 165×2 doubles (ulisttot_r/i) × n_atoms
plus 56 doubles (blist). Per-neighbour state ≈ 165×2 doubles (ulist_ij) × total
neighbours (~50 per atom × 2000 atoms = ~100k pairs × 2.6 KB = ~260 MB). Per-
neighbour derivatives (dulist_r/i) same shape. Total device memory at 2000-
atom fixture: **~520 MB persistent + per-call scratch**. Fits comfortably on
24 GB RTX 5080 but not a toy.

EAM GPU precedent (`src/gpu/eam_alloy_gpu.cu` 1215 lines) covered
density→embedding→force — far simpler state machine than SNAP's five-pass
bispectrum (Ui → Zi → Bi → Yi → deidrj per neighbour × duidrj).

**Honest estimate:** full T8.6 is a 2+ day task, not single-session.

### T8.6b — actual kernel body (NEXT SESSION)

Full port of SnaEngine to device. Structural approach:

- **Kernel decomposition** — three separate kernels per `compute()`:
  1. `snap_ui_kernel<<<n_atoms, warp>>>` — Pass 1: compute Wigner U functions,
     accumulate to `ulisttot`. One atom per thread-block, warp-level reduction
     across neighbours.
  2. `snap_yi_kernel<<<n_atoms, warp>>>` — Pass 2: compute Z list, contract
     with β → Y list.
  3. `snap_deidrj_kernel<<<n_atoms * n_neigh_avg, warp>>>` — Pass 3: per-
     neighbour dE/dr using Clebsch-Gordan chain rule, atomicAdd to forces.
- **Memory layout** — flatten all `T**` pointer-arrays to 1D contiguous
  `T*` device buffers. Row-major: `ulisttot[atom * idxu_max + u_idx]`.
  `idxcg_block[j1][j2][j]` → `idxcg_block[((j1 * jdim) + j2) * jdim + j]`
  linear. Index tables are read-only → upload once in `init()`, reuse across
  compute() calls (same pattern as EAM spline cache T6.9a).
- **Bit-exactness** — mandatory per D-M6-7 / D-M8-8. All FP summation inside
  compute_ui / compute_zi / compute_yi must hit operands in the same order as
  CPU (neighbour-loop + m1/m2 nested loops). Challenge: neighbour loop on
  CPU is sequential per atom; warp-parallel version on GPU requires
  **explicit per-thread partitioning + canonical reduction via Kahan** (the
  D-M6-15 canonical reduction pattern). This is load-bearing work.
- **Attribution** — every new `.cu`/`.cuh` file reproduces the LAMMPS GPLv2
  block verbatim (same pattern as `src/potentials/snap/sna_engine.cpp`). The
  device port is "this code was translated to CUDA from upstream" — still
  GPLv2.
- **Tests** — `tests/gpu/test_snap_gpu_functional.cpp` single-atom W
  bispectrum correctness; `tests/gpu/test_snap_gpu_bit_exact.cpp` GPU vs
  CPU on 64-atom W ≤ 1e-12 rel; `tests/gpu/test_nvtx_audit.cpp` extended to
  snap_gpu.cu.

## Invariants to preserve

- **D-M6-7 bit-exact** (Reference GPU ≡ Reference CPU ≤ 1e-12 rel force +
  ≤ 1e-12 rel PE). Extends to SNAP as D-M8-13 chain at T8.7.
- **LAMMPS GPLv2 attribution** — every ported TU carries the upstream
  license block verbatim (T8.4a/b precedent).
- **`__restrict__`** on all pointer params per master spec §D.16. No
  exceptions without explicit `NOLINT(tdmd-missing-restrict)` rationale.
- **TDMD_NVTX_RANGE** wrapping every `<<<...>>>` launch per gpu/SPEC §9 and
  T6.11 audit. `test_nvtx_audit` will fail on any unwrapped launch.
- **Ownership** — `SnapGpu::compute()` writes to atoms.f buffers only; no
  mutation of positions, neighbors, or SnapData.
- **Reference path sacred** — `Fp64ReferenceBuild + Reference ExecProfile`
  MUST remain the oracle. MixedFast variant is T8.9 separate work.
- **PIMPL firewall** (D-M6-17) — public `.hpp` is CUDA-free; all
  `__device__` / `__global__` contained in `.cu` body.

## Explicit assumptions

1. M8 parser already rejects chemflag/switchinnerflag/quadraticflag (T8.4a —
   confirmed via reading `snap_file.cpp`). GPU port inherits these limitations;
   attempting to use them throws at `SnapGpu::compute()` not at kernel launch.
2. T6 fixture (W_2940, twojmax=8, 56 linear coefficients) is the only
   coefficient set exercised in M8 tests. GPU port is not exercised for
   larger twojmax until M9+ MEAM chain.
3. Cell-grid neighbour list is same `NeighborListGpu` that EAM uses (T6.4).
   Half-list → full-list bridge moves to device inside SnapGpu (cannot do
   it on host because then we'd pay the H2D trip twice).
4. FP64 only in T8.6. MixedFast SNAP kernel is T8.9 (separate BuildFlavor
   `MixedFastSnapOnlyBuild` established at T8.8).

## Risks

- **R-T8.6a-1 — adapter might cause link-ordering cycles.** EAM adapter
  pattern already resolved similar gpu↔potentials dep. Mitigation: follow
  EamAlloyGpuAdapter structure 1:1; data-oblivious gpu/ per module SPEC §1.1.
- **R-T8.6a-2 — CI compile-gpu job fails on new snap_gpu.cu if nvcc trips
  on an unexpected header.** Mitigation: keep snap_gpu.cu body minimal
  (just the stub throw) — no `__device__` code in T8.6a; full nvcc stress
  test deferred to T8.6b.
